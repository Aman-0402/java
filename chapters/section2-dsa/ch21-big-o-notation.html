Chapter 21: Big O Notation

Big O notation is the universal language programmers use to describe how an algorithm's performance scales as the input grows larger.

📜 History: Big O notation was formalized by German mathematician Paul Bachmann in 1894 — over 100 years before modern computers! Computer scientists adopted it in the 1960s to analyze algorithm efficiency. It's a mathematical concept older than the transistor.

Why Performance Scaling Matters

Imagine you need to fill a container with water. Filling a small glass takes a second. Filling a swimming pool takes hours — not because your tap got slower, but because the size of the problem changed. The same idea applies to algorithms: some grow slowly as input grows, others grow explosively.

Big O notation captures that growth rate. It answers the question: "If I double the input size, what happens to the runtime?" It does not tell you the exact number of milliseconds — it tells you the shape of the curve.

ℹ️ Big O focuses on the dominant term as input size (n) approaches infinity. Constants and smaller terms are dropped because they become irrelevant at scale.

O(1) — Constant Time

No matter how big the input, the operation always takes the same amount of work. Think of looking up a word if you already know the exact page number — it doesn't matter how thick the book is.

// O(1) — accessing an array by index is always one step
int[] numbers = {10, 20, 30, 40, 50};
int first = numbers[0];  // instant, no matter how large the array

O(log n) — Logarithmic Time

Each step cuts the problem roughly in half. Binary search is the classic example: to find a name in a sorted phone book, you open to the middle, discard half, repeat. A phone book with 1,000,000 names only takes about 20 comparisons.

// O(log n) — binary search
int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

🔍 How This Code Works

  • int left = 0, right = arr.length - 1 — initializes the search window to cover the entire array
  • while (left <= right) — loop continues as long as the search space has at least one element
  • int mid = left + (right - left) / 2 — safe mid-point calculation that avoids integer overflow (avoids (left+right)/2 which can overflow)
  • if (arr[mid] == target) return mid — found the target; return its index immediately
  • left = mid + 1 — target must be in the right half; discard everything up to and including mid
  • right = mid - 1 — target must be in the left half; discard everything from mid onward
  • return -1 — loop ended without a match; target is not in the array
  • Each iteration halves the search space, so the loop runs at most log₂(n) times — that is why this is O(log n)

O(n) — Linear Time

Work grows in direct proportion to input size. To find the maximum value in an unsorted list you must look at every element once. Double the list, double the work.

// O(n) — scanning every element once
int findMax(int[] arr) {
    int max = arr[0];
    for (int num : arr) {      // visits every element
        if (num > max) max = num;
    }
    return max;
}

O(n²) — Quadratic Time

For every element you do work proportional to n again — a loop inside a loop. Bubble sort is the textbook example. Double the input, quadruple the work.

// O(n²) — bubble sort (nested loops)
void bubbleSort(int[] arr) {
    int n = arr.length;
    for (int i = 0; i < n - 1; i++) {          // outer loop
        for (int j = 0; j < n - i - 1; j++) {  // inner loop
            if (arr[j] > arr[j + 1]) {
                int temp = arr[j];
                arr[j] = arr[j + 1];
                arr[j + 1] = temp;
            }
        }
    }
}

🔍 How This Code Works

  • for (int i = 0; i < n - 1; i++) — outer loop runs n-1 passes; after each pass one more element is in its final position at the right end
  • for (int j = 0; j < n - i - 1; j++) — inner loop only checks unsorted elements; n - i - 1 shrinks on each pass because the last i elements are already sorted
  • if (arr[j] > arr[j + 1]) — compares each pair of adjacent elements
  • int temp = arr[j]; arr[j] = arr[j+1]; arr[j+1] = temp — classic three-variable swap; the larger element "bubbles" one position to the right
  • The nested loops together perform roughly n²/2 comparisons — hence O(n²)
  • After pass 1 the largest element is at the end; after pass 2 the second-largest is in place, and so on
🔍 Fun Fact: Google's original PageRank algorithm ran in O(n log n) time. If they had used an O(n²) algorithm, ranking 1 billion pages would have taken 1,000x longer. Choosing the right Big O complexity is why Google Search returns results in milliseconds, not hours.

Here is how each complexity class behaves as n grows. Notice how O(n²) explodes while O(log n) barely moves.

Why We Drop Constants and Lower-Order Terms

An algorithm that runs 3n + 5 operations is still O(n). At n = 1,000,000 the constant 3 and the +5 are irrelevant compared to n itself. Big O captures the growth pattern, not the exact count.

// This is O(n), not O(3n + 5)
void example(int[] arr) {
    int sum = 0;         // 1 operation
    for (int x : arr) { // n operations
        sum += x;        // 1 op each
        sum *= 2;        // 1 op each
        sum -= 1;        // 1 op each
    }
    System.out.println(sum); // 1 operation
    // Total: 3n + 2 → still O(n)
}
💡 When analysing your own code, count the loops. One loop → O(n). A loop inside a loop → O(n²). No loop → O(1). This heuristic handles 90% of interview situations.

Best, Worst, and Average Case

Big O typically describes the worst case — the scenario that takes the longest. For linear search on an array of n elements:

  • Best case: target is the first element → O(1)
  • Average case: target is somewhere in the middle → O(n/2) → O(n)
  • Worst case: target is last or missing → O(n)

When people say "linear search is O(n)" they mean the worst case, which is the safe guarantee to design around.

🌍 Real World: When you load your Instagram feed, an O(log n) algorithm ranks thousands of posts. If Instagram used O(n²), loading your feed with 10,000 potential posts would be 100 million operations — your phone would freeze for seconds every refresh.
⚠️ Do not confuse O(log n) with "fast" and O(n²) with "slow" in absolute terms. For n = 5, bubble sort is perfectly fine. Big O matters when n is large — hundreds of thousands or more.

🧠 Multiple Choice Questions

1. What is the Big O complexity of binary search?

  • A) O(n)
  • B) O(n²)
  • C) O(log n)
  • D) O(1)
Show Answer

✅ C) O(log n) — Binary search halves the search space each step. For n=1,000,000 elements, it takes at most 20 comparisons.

2. Which Big O represents constant time (doesn't depend on input size)?

  • A) O(n)
  • B) O(1)
  • C) O(log n)
  • D) O(0)
Show Answer

✅ B) O(1) — Accessing array[5] takes the same time whether the array has 10 or 10 million elements.

3. Sorted from fastest to slowest, which order is correct?

  • A) O(1) < O(log n) < O(n) < O(n log n) < O(n²)
  • B) O(log n) < O(1) < O(n) < O(n²)
  • C) O(n) < O(log n) < O(1) < O(n²)
  • D) O(1) < O(n) < O(log n) < O(n²)
Show Answer

✅ A) O(1) < O(log n) < O(n) < O(n log n) < O(n²) — This is the standard complexity hierarchy from most to least efficient.

4. What is the Big O of accessing an element by index in an ArrayList?

  • A) O(n)
  • B) O(log n)
  • C) O(n²)
  • D) O(1)
Show Answer

✅ D) O(1) — ArrayList is backed by an array. Index access = one memory lookup = constant time.
