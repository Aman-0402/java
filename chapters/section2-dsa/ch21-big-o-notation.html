Chapter 21: Big O Notation

Big O notation is the universal language programmers use to describe how an algorithm's performance scales as the input grows larger.

Why Performance Scaling Matters

Imagine you need to fill a container with water. Filling a small glass takes a second. Filling a swimming pool takes hours — not because your tap got slower, but because the size of the problem changed. The same idea applies to algorithms: some grow slowly as input grows, others grow explosively.

Big O notation captures that growth rate. It answers the question: "If I double the input size, what happens to the runtime?" It does not tell you the exact number of milliseconds — it tells you the shape of the curve.

ℹ️ Big O focuses on the dominant term as input size (n) approaches infinity. Constants and smaller terms are dropped because they become irrelevant at scale.

O(1) — Constant Time

No matter how big the input, the operation always takes the same amount of work. Think of looking up a word if you already know the exact page number — it doesn't matter how thick the book is.

// O(1) — accessing an array by index is always one step
int[] numbers = {10, 20, 30, 40, 50};
int first = numbers[0];  // instant, no matter how large the array

O(log n) — Logarithmic Time

Each step cuts the problem roughly in half. Binary search is the classic example: to find a name in a sorted phone book, you open to the middle, discard half, repeat. A phone book with 1,000,000 names only takes about 20 comparisons.

// O(log n) — binary search
int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

O(n) — Linear Time

Work grows in direct proportion to input size. To find the maximum value in an unsorted list you must look at every element once. Double the list, double the work.

// O(n) — scanning every element once
int findMax(int[] arr) {
    int max = arr[0];
    for (int num : arr) {      // visits every element
        if (num > max) max = num;
    }
    return max;
}

O(n²) — Quadratic Time

For every element you do work proportional to n again — a loop inside a loop. Bubble sort is the textbook example. Double the input, quadruple the work.

// O(n²) — bubble sort (nested loops)
void bubbleSort(int[] arr) {
    int n = arr.length;
    for (int i = 0; i < n - 1; i++) {          // outer loop
        for (int j = 0; j < n - i - 1; j++) {  // inner loop
            if (arr[j] > arr[j + 1]) {
                int temp = arr[j];
                arr[j] = arr[j + 1];
                arr[j + 1] = temp;
            }
        }
    }
}

Growth Rate Comparison

Here is how each complexity class behaves as n grows. Notice how O(n²) explodes while O(log n) barely moves.

Why We Drop Constants and Lower-Order Terms

An algorithm that runs 3n + 5 operations is still O(n). At n = 1,000,000 the constant 3 and the +5 are irrelevant compared to n itself. Big O captures the growth pattern, not the exact count.

// This is O(n), not O(3n + 5)
void example(int[] arr) {
    int sum = 0;         // 1 operation
    for (int x : arr) { // n operations
        sum += x;        // 1 op each
        sum *= 2;        // 1 op each
        sum -= 1;        // 1 op each
    }
    System.out.println(sum); // 1 operation
    // Total: 3n + 2 → still O(n)
}
💡 When analysing your own code, count the loops. One loop → O(n). A loop inside a loop → O(n²). No loop → O(1). This heuristic handles 90% of interview situations.

Best, Worst, and Average Case

Big O typically describes the worst case — the scenario that takes the longest. For linear search on an array of n elements:

  • Best case: target is the first element → O(1)
  • Average case: target is somewhere in the middle → O(n/2) → O(n)
  • Worst case: target is last or missing → O(n)

When people say "linear search is O(n)" they mean the worst case, which is the safe guarantee to design around.

⚠️ Do not confuse O(log n) with "fast" and O(n²) with "slow" in absolute terms. For n = 5, bubble sort is perfectly fine. Big O matters when n is large — hundreds of thousands or more.
