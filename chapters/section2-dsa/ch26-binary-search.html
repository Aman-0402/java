Chapter 26: Binary Search

Binary search cuts a sorted array in half each time, finding any value in O(log n) steps instead of scanning every element.

📜 History: Binary search's concept dates back to 1946, described by John Mauchly (co-inventor of ENIAC, the first electronic computer). But the first bug-free implementation wasn't published until 1962 by D.H. Lehmer. The algorithm seemed simple but had a subtle integer overflow bug in mid = (low + high) / 2 — which Java programmer Joshua Bloch discovered in Java's Arrays.binarySearch() in 2006 and fixed to mid = low + (high - low) / 2.

The Phone Book Analogy

Imagine you are looking for "Smith" in a phone book. You would never start at page one and read every name. Instead, you open the book in the middle. If the middle page shows "Miller", you know Smith is in the right half — so you discard the left half entirely and repeat on the remaining pages. That is binary search.

The key requirement: the collection must already be sorted. Binary search cannot work on a random pile of names any more than you could use the "open the middle" trick on a shuffled deck of cards.

Linear Search vs Binary Search

Linear search checks every element from left to right until it finds the target or runs out of elements. It works on unsorted data but is slow for large arrays — up to n comparisons in the worst case (O(n)).

Binary search eliminates half the remaining elements with each comparison. An array of 1,000,000 elements needs at most 20 comparisons (because 220 > 1,000,000). That is O(log n).

Algorithm Steps

Three pointers track the search space: low (start index), high (end index), and mid (middle index).

  1. Set low = 0 and high = array.length - 1.
  2. While low <= high:
  3.   Calculate mid = low + (high - low) / 2.
  4.   If array[mid] == target → found, return mid.
  5.   If array[mid] < target → target is in the right half, set low = mid + 1.
  6.   If array[mid] > target → target is in the left half, set high = mid - 1.
  7. If the loop ends without a match, return -1 (not found).

Java Implementation — Iterative

public class BinarySearch {

    public static int binarySearch(int[] arr, int target) {
        int low = 0;
        int high = arr.length - 1;

        while (low <= high) {
            // Safe mid: avoids integer overflow when low+high is large
            int mid = low + (high - low) / 2;

            if (arr[mid] == target) {
                return mid;           // found — return the index
            } else if (arr[mid] < target) {
                low = mid + 1;        // target is in the right half
            } else {
                high = mid - 1;       // target is in the left half
            }
        }

        return -1;  // target not in array
    }

    public static void main(String[] args) {
        int[] sorted = {1, 3, 5, 7, 9, 11, 13};

        System.out.println(binarySearch(sorted, 7));   // 3
        System.out.println(binarySearch(sorted, 11));  // 5
        System.out.println(binarySearch(sorted, 13));  // 6
        System.out.println(binarySearch(sorted, 4));   // -1 (not found)
    }
}

Java Implementation — Recursive

public class BinarySearchRecursive {

    public static int binarySearch(int[] arr, int target, int low, int high) {
        // Base case: search space is empty
        if (low > high) {
            return -1;
        }

        int mid = low + (high - low) / 2;

        if (arr[mid] == target) {
            return mid;
        } else if (arr[mid] < target) {
            // Search the right half
            return binarySearch(arr, target, mid + 1, high);
        } else {
            // Search the left half
            return binarySearch(arr, target, low, mid - 1);
        }
    }

    public static void main(String[] args) {
        int[] sorted = {1, 3, 5, 7, 9, 11, 13};

        // Pass the full range on the first call
        int result = binarySearch(sorted, 7, 0, sorted.length - 1);
        System.out.println("Found at index: " + result);  // 3

        int missing = binarySearch(sorted, 4, 0, sorted.length - 1);
        System.out.println("Found at index: " + missing); // -1
    }
}

Built-in: Arrays.binarySearch()

Java's standard library ships a ready-to-use binary search. It returns the index of the target if found, or a negative value if not found. The negative value encodes where the element would be inserted to maintain sorted order. 

import java.util.Arrays;
import java.util.Collections;
import java.util.List;

public class BuiltInBinarySearch {
    public static void main(String[] args) {
        int[] sorted = {1, 3, 5, 7, 9, 11, 13};

        // Returns the index (3) when found
        int found = Arrays.binarySearch(sorted, 7);
        System.out.println("Found at: " + found);   // 3

        // Returns a negative number when not found
        int missing = Arrays.binarySearch(sorted, 4);
        System.out.println("Not found: " + missing); // negative

        // Also works on sorted List
        List<Integer> list = List.of(1, 3, 5, 7, 9, 11, 13);
        int idx = Collections.binarySearch(list, 11);
        System.out.println("List search: " + idx);   // 5
    }
}
💡 Tip — Safe Mid Calculation: Always compute mid = low + (high - low) / 2 instead of (low + high) / 2. When both low and high are large integers their sum can overflow a 32-bit int. The safe formula prevents that subtle bug. This was actually a bug in Java's own Arrays.binarySearch() for years before it was fixed.
⚠️ Common Mistake — Searching an Unsorted Array: Binary search gives silently wrong answers on unsorted data — it will not throw an exception or warn you. Always sort the array first with Arrays.sort(arr). If you need to search an unsorted collection use linear search instead; binary search on unsorted data is worse than useless because you cannot trust the result.
🔍 Fun Fact: Binary search is so efficient that searching 1 billion sorted items takes at most 30 comparisons (log₂(1,000,000,000) ≈ 30). A linear search of the same dataset would average 500 million comparisons. This difference is why sorted databases can answer queries in milliseconds.

Why O(log n)?

Each step halves the search space. Starting from n elements: 

After 1 step:  n / 2    elements remain
After 2 steps: n / 4    elements remain
After 3 steps: n / 8    elements remain
After k steps: n / 2^k  elements remain

Search ends when n / 2^k = 1
Solving: k = log₂(n)

Examples:
  n = 10        →  k ≤  4 comparisons
  n = 1,000     →  k ≤ 10 comparisons
  n = 1,000,000 →  k ≤ 20 comparisons
  n = 1,000,000,000 → k ≤ 30 comparisons

A billion-element array, 30 comparisons. That is the power of logarithmic growth.

🌍 Real World: Every database index uses binary search (or its cousin, B-tree search). When you search for a username on any website, the database performs a binary search on an indexed column. Without indexes (and binary search), every query would scan the entire table — millions of rows for a single lookup.
