Chapter 29: HashMap Internals

HashMap gives you O(1) average-case lookups by converting keys into array indices — understanding how it does this (and when it breaks down) makes you a far more effective Java developer.

📜 History: Hash functions were invented by Hans Peter Luhn at IBM in 1953. The idea of using a hash to index into an array (hash table) was published by H.P. Luhn in 1953 and later by Arnold Dumey. Java's HashMap (introduced in Java 2, 1998) is a refined implementation of this 70-year-old concept.

The Problem HashMap Solves

Suppose you want to look up a student's grade by their name. Using a plain array you would have to scan every element — O(n). A sorted array with binary search gives O(log n). But what if you could jump directly to the right answer in one step? That is what HashMap achieves: O(1) average-case lookup by converting the key into an array index.

The Core Idea: Array of Buckets

Internally, a HashMap is an array of buckets. Each bucket can hold one or more key-value pairs. When you call map.put(key, value), Java:

  1. Calls key.hashCode() to get an integer hash.
  2. Reduces the hash to a valid array index: index = hash % buckets.length.
  3. Stores the key-value pair in the bucket at that index.

When you call map.get(key), Java repeats the same two steps to compute the index and jumps straight to that bucket — no scanning needed.

Hash Codes

Every Java object inherits a hashCode() method from Object. The contract is:

  • If a.equals(b) is true, then a.hashCode() == b.hashCode() must be true.
  • If a.hashCode() == b.hashCode(), a.equals(b) may or may not be true (that is a collision).

Java's built-in types like String, Integer, and Long have well-implemented hashCode() methods. The String hash is computed from every character: 

// String hashCode — roughly:
// hash = s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]

System.out.println("Alice".hashCode());  // some integer, always the same
System.out.println("Alice".hashCode());  // same integer again — consistent

Collision Handling: Chaining

Two different keys can produce the same bucket index — that is a collision. Java's HashMap handles collisions by chaining: each bucket holds a linked list (or a red-black tree in Java 8+ when the chain grows long) of all the key-value pairs that hashed to that index. When searching, Java scans the short list in that bucket using equals() to find the right key. 

Chaining in bucket 6 (from diagram above):

  bucket[6] ──→ Node("Bob", 85) ──→ Node("Carol", 88) ──→ null

get("Carol"):
  1. Compute index: hash("Carol") % 8 = 6
  2. Go to bucket[6]
  3. Check "Bob".equals("Carol") → false, move to next node
  4. Check "Carol".equals("Carol") → true → return 88

Load Factor and Rehashing

As more entries are added, buckets fill up and chains grow longer, slowing down lookups. The load factor controls when the internal array is resized.

Default load factor: 0.75. This means: when the number of entries exceeds 0.75 × buckets.length, the HashMap doubles its bucket array and rehashes all existing entries into the new larger array. Rehashing is O(n) work, but it happens rarely enough that amortised across all insertions the cost is still O(1) per put. 

Example with initial capacity = 4, load factor = 0.75:

  Threshold = 4 × 0.75 = 3 entries

  Add entry 1 → 1 entry  (threshold not hit)
  Add entry 2 → 2 entries (threshold not hit)
  Add entry 3 → 3 entries (threshold = 3 → REHASH!)
    → New capacity = 8, all entries re-indexed

  Add entry 4 → 4 entries (new threshold = 8 × 0.75 = 6)
  ...
  Add entry 6 → 6 entries → REHASH again!
    → New capacity = 16
🔍 Fun Fact: Java's HashMap has an interesting quirk: if many keys hash to the same bucket (collision), Java 8+ converts that bucket's linked list into a Red-Black Tree when it has more than 8 entries. This changes worst-case lookup from O(n) to O(log n) — a subtle optimization that most Java developers don't know about.
💡 Tip — Pre-size Your HashMap: If you know you will insert roughly n entries, construct the map with new HashMap<>(n / 0.75 + 1) to avoid rehashing. Rehashing copies every entry — avoiding it is a free performance win for large maps.

Custom Class as a HashMap Key

Using your own class as a HashMap key is a common interview topic and a common production bug. If two objects are logically equal but have different default hashCode() values, the HashMap will put them in different buckets and never find the second one when you search by the first.

The rule: whenever you override equals(), you must also override hashCode(). 

import java.util.HashMap;
import java.util.Objects;

public class Student {
    String name;
    int id;

    public Student(String name, int id) {
        this.name = name;
        this.id = id;
    }

    // Two students are equal if they have the same id
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (!(obj instanceof Student)) return false;
        Student other = (Student) obj;
        return this.id == other.id;
    }

    // hashCode MUST be consistent with equals
    // If two objects are equal their hashCodes must be equal too
    @Override
    public int hashCode() {
        return Objects.hash(id);  // base hashCode only on fields used in equals
    }

    public static void main(String[] args) {
        HashMap<Student, Integer> grades = new HashMap<>();

        Student s1 = new Student("Alice", 101);
        grades.put(s1, 95);

        // Create a NEW object with the same id
        Student s2 = new Student("Alice", 101);

        // With equals+hashCode correctly overridden:
        System.out.println(grades.get(s2));  // 95  ← found correctly

        // Without overriding (using default Object identity):
        // grades.get(s2) would return null because s1 != s2 in memory
    }
}

🔍 How This Code Works

  • @Override public boolean equals(Object obj) — defines logical equality; two Student objects are equal if they share the same id, regardless of being different objects in memory
  • if (this == obj) return true — fast path: the same object reference is trivially equal to itself
  • if (!(obj instanceof Student)) return false — guards against comparing a Student with a completely different type
  • return Objects.hash(id) — generates a hash based only on the fields used in equals(); using id here means two students with the same id will hash to the same bucket
  • grades.get(s2) — works correctly because HashMap first calls s2.hashCode() to find the bucket, then calls s2.equals(s1) to confirm the match; both methods are consistent so the lookup succeeds
  • Without overriding, the default hashCode() is based on object identity (memory address), so s1 and s2 would hash to different buckets and get(s2) would return null
  • The contract rule: if a.equals(b) is true, then a.hashCode() == b.hashCode() must also be true — violating this breaks all hash-based collections
🌍 Real World: DNS resolution (converting "google.com" to an IP address) uses a giant distributed hash table. Your computer caches DNS results in a local HashMap (domain → IP). When you type a URL, Java's InetAddress.getByName() checks this cache first — a HashMap lookup that takes nanoseconds before making a network request that takes milliseconds.
⚠️ Common Mistake — Mutable Keys: Never mutate an object while it is being used as a HashMap key. The hashCode changes when the object changes, so the map can no longer find the entry (it looks in the wrong bucket). Use immutable objects — like String, Integer, or records — as keys whenever possible.

A Quick Demo of HashMap Usage

import java.util.HashMap;
import java.util.Map;

public class HashMapDemo {
    public static void main(String[] args) {
        HashMap<String, Integer> wordCount = new HashMap<>();

        String[] words = {"apple", "banana", "apple", "cherry", "banana", "apple"};

        for (String word : words) {
            // getOrDefault avoids a NullPointerException on first occurrence
            wordCount.put(word, wordCount.getOrDefault(word, 0) + 1);
        }

        System.out.println(wordCount);
        // {apple=3, banana=2, cherry=1}  (order may vary)

        // Iterate over entries
        for (Map.Entry<String, Integer> entry : wordCount.entrySet()) {
            System.out.println(entry.getKey() + " appears " + entry.getValue() + " times");
        }
    }
}

🔍 How This Code Works

  • wordCount.getOrDefault(word, 0) — returns the current count for word, or 0 if the key does not yet exist; this avoids a NullPointerException on the very first occurrence of each word
  • wordCount.put(word, ... + 1) — either inserts a new entry with count 1 (first occurrence) or overwrites the existing count with an incremented value
  • This is the standard frequency-counting idiom in Java; the equivalent with merge() is wordCount.merge(word, 1, Integer::sum)
  • wordCount.entrySet() — returns a Set of Map.Entry objects, each holding one key-value pair; this is the most efficient way to iterate both keys and values simultaneously
  • entry.getKey() and entry.getValue() — retrieve the key and value from each map entry without a second lookup
  • HashMap iteration order is not guaranteed to match insertion order — use LinkedHashMap if you need insertion-order iteration
