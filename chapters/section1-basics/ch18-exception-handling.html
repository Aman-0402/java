📗 Section 1 — Java Basics

Chapter 18: Exception Handling

Exceptions are runtime errors that would crash your program — exception handling lets you catch them gracefully and keep the program running.

What Is an Exception?

Imagine you're following a recipe and step 3 says "add two eggs". You open the fridge and find no eggs. The recipe can't continue as written. You have a choice: crash (throw the recipe book across the kitchen) or handle it gracefully (substitute a different ingredient, or ask someone to buy eggs).

In Java, an exception is an unexpected event that happens while a program is running — a file that doesn't exist, a number divided by zero, or invalid user input. Without exception handling, Java stops the program and prints a stack trace. With it, you can respond gracefully.

🔍 Fun Fact: The most common exception in Java production systems is NullPointerException (NPE). Tony Hoare, who invented the null reference in 1965, called it his "billion-dollar mistake" in a 2009 talk. Null references have caused more bugs, crashes, and security vulnerabilities than almost any other single programming concept.

Common Exceptions You Will Encounter

  • NullPointerException — calling a method on a null reference.
  • ArrayIndexOutOfBoundsException — accessing an array index that doesn't exist.
  • NumberFormatException — converting a non-numeric String (e.g. "abc") to an integer.
  • ArithmeticException — dividing an integer by zero.
  • ClassCastException — casting an object to an incompatible type.

try-catch-finally

Wrap risky code in a try block. If an exception occurs, Java jumps to the matching catch block. The finally block (optional) always runs, whether or not an exception occurred — useful for cleanup like closing files. 

public class Main {
    public static void main(String[] args) {
        try {
            int result = 10 / 0;   // ArithmeticException!
            System.out.println(result);
        } catch (ArithmeticException e) {
            System.out.println("Cannot divide by zero: " + e.getMessage());
        } finally {
            System.out.println("This always runs.");
        }
    }
}

Output:

Cannot divide by zero: / by zero
This always runs.

Catching Multiple Exception Types

You can stack multiple catch blocks, or combine them in one block using the pipe | operator (Java 7+): 

public class Main {
    public static void main(String[] args) {
        String input = "hello";

        try {
            int number = Integer.parseInt(input);   // NumberFormatException
            int[] arr  = new int[5];
            arr[10]    = number;                    // ArrayIndexOutOfBoundsException
        } catch (NumberFormatException e) {
            System.out.println("Not a valid number: " + e.getMessage());
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Array index out of bounds: " + e.getMessage());
        }
    }
}
⚠️ Common mistake: Catching the broad Exception class to "handle everything". This hides real bugs because you can no longer tell what went wrong. Always catch the most specific exception type you expect.
🌍 Real World: In 2012, a bug in error handling code at Knight Capital Group caused a trading algorithm to lose $440 million in 45 minutes. The faulty exception handling went undetected in testing. This is why robust exception handling in financial software is not optional — it is mission-critical.

Checked vs Unchecked Exceptions

Java has two categories of exceptions:

  • Checked exceptions — the compiler forces you to handle them (use try-catch or declare throws). Example: IOException when reading a file.
  • Unchecked exceptions — subclasses of RuntimeException. The compiler does not force handling. Example: NullPointerException, ArithmeticException.

throw and throws

Use throw to manually trigger an exception. Use throws in a method signature to tell callers that this method might throw a checked exception they must handle. 

public class AgeValidator {
    // 'throws' warns callers this method can throw
    public static void validateAge(int age) throws IllegalArgumentException {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("Invalid age: " + age);
        }
        System.out.println("Age " + age + " is valid.");
    }

    public static void main(String[] args) {
        try {
            validateAge(25);    // OK
            validateAge(-5);    // throws exception
        } catch (IllegalArgumentException e) {
            System.out.println("Error: " + e.getMessage());
        }
    }
}

Output:

Age 25 is valid.
Error: Invalid age: -5

Custom Exceptions (Brief)

You can create your own exception type by extending Exception (checked) or RuntimeException (unchecked): 

// Custom checked exception
public class InsufficientFundsException extends Exception {
    public InsufficientFundsException(double amount) {
        super("Insufficient funds. Short by: $" + amount);
    }
}

// Usage
public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException(amount - balance);
    }
    balance -= amount;
}
💡 Tip: The exception message passed to super() is what e.getMessage() returns in the catch block. Make your messages descriptive — they are your debugging lifeline.
📜 History: Java's checked exceptions — exceptions you must declare with throws — were James Gosling's idea to force developers to handle errors explicitly. In hindsight, the Java community debates this choice: most modern languages (Kotlin, C#, Python) abandoned checked exceptions because developers often bypass them with empty catch blocks.
ℹ️ Stack trace reading: When an uncaught exception crashes your program, Java prints a stack trace. Read it from top to bottom — the top line shows the exception type and message; the lines below show which methods were called to get there. The first line mentioning your class is usually where the bug lives.

🧠 Multiple Choice Questions

1. Which block always executes whether or not an exception is thrown?

  • A) catch
  • B) try
  • C) finally
  • D) throws
Show Answer

✅ C) finally — The finally block runs regardless of whether an exception occurred, making it ideal for cleanup code (closing files, releasing connections).

2. What is the parent class of all Java exceptions?

  • A) Error
  • B) RuntimeException
  • C) Throwable
  • D) Exception
Show Answer

✅ C) Throwable — Both Exception and Error extend Throwable. Errors (like OutOfMemoryError) are serious JVM issues you typically don't catch.

3. What is a checked exception?

  • A) An exception that Java checks for security
  • B) An exception that must be either caught or declared with throws
  • C) A RuntimeException
  • D) An exception with a checkbox UI
Show Answer

✅ B) An exception that must be either caught or declared with throws — Checked exceptions (like IOException) must be handled. Unchecked (like NullPointerException) don't have to be.

4. What does throw new IllegalArgumentException("msg") do?

  • A) Catches an exception
  • B) Declares an exception
  • C) Manually creates and throws an exception at that point in code
  • D) Logs an error
Show Answer

✅ C) Manually creates and throws an exception at that point in code — You use throw to signal that something went wrong in your own code.
