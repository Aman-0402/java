Chapter 16: Encapsulation and Abstraction

Encapsulation hides the data inside an object; abstraction hides the complexity of how something works — together they make code safer and easier to use.

Encapsulation — Protecting Your Data

Think of a capsule medicine tablet. The drug inside is protected by an outer shell. You interact with the tablet by swallowing it — you never directly touch the drug inside. The shell controls how and when the drug is released.

In Java, encapsulation means keeping an object's fields private so outside code cannot change them directly. Instead, you provide controlled access through public getter and setter methods.

Why Private Fields Matter

Without encapsulation anyone can write: 

Person p = new Person();
p.age = -50;    // Invalid age — no protection!

With encapsulation the setter can reject bad values: 

public void setAge(int age) {
    if (age >= 0 && age <= 150) {
        this.age = age;
    } else {
        System.out.println("Invalid age!");
    }
}

Getters and Setters

The naming convention is getFieldName() for reading and setFieldName(value) for writing. For boolean fields, the getter is named isFieldName(). 

public class BankAccount {
    private String owner;
    private double balance;

    public BankAccount(String owner, double initialBalance) {
        this.owner   = owner;
        this.balance = initialBalance;
    }

    // Getter — read-only access to owner
    public String getOwner() {
        return owner;
    }

    // Getter for balance
    public double getBalance() {
        return balance;
    }

    // No direct setter for balance — use controlled methods instead
    public void deposit(double amount) {
        if (amount > 0) {
            balance += amount;
        }
    }

    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
        } else {
            System.out.println("Insufficient funds or invalid amount.");
        }
    }

    public static void main(String[] args) {
        BankAccount acc = new BankAccount("Alice", 1000.0);
        acc.deposit(500.0);
        acc.withdraw(200.0);
        System.out.println(acc.getOwner() + "'s balance: $" + acc.getBalance());
        // acc.balance = 999999;  // compile error — private!
    }
}

Output:

Alice's balance: $1300.0
💡 Tip: Make all fields private by default. Only expose what is truly needed through getters and setters. This gives you the freedom to change the internal implementation later without breaking any code that uses your class.
🌍 Real World: Your banking app uses encapsulation to protect your balance. The balance field is private — you can only access it through getBalance(). The withdraw() method checks if you have sufficient funds before reducing the balance. Without encapsulation, any part of the code could set balance = 0 directly.
⚠️ Common mistake: Making everything public "to keep it simple". This removes all protection and makes your code fragile — any part of the program can corrupt your object's data.
🔍 Fun Fact: The most violated principle in beginner code is encapsulation — making fields public instead of private. In a 2021 survey of 10,000 GitHub Java repositories, over 60% of beginner projects had at least one public field that should have been private. It remains one of the most common findings in professional code reviews.

Abstraction — Hiding Complexity

When you drive a car, you press the accelerator. You do not need to know the fuel injection sequence, combustion timing, or transmission ratios. The car's design abstracts away those details — you get a simple interface (pedal, wheel, brake) that hides the engine's complexity.

In Java, abstraction is achieved with abstract classes and interfaces. An abstract class defines what something must do, without specifying how.

Abstract Classes and Methods

An abstract method has no body — just a declaration. Concrete subclasses must provide the body. A class with at least one abstract method must itself be declared abstract, and you cannot instantiate it directly. 

// Abstract class — cannot be instantiated directly
public abstract class Shape {
    // Abstract method — no body, subclass must implement
    public abstract double area();

    // Concrete method — available to all subclasses
    public void printArea() {
        System.out.println("Area: " + area());
    }
}

public class Circle extends Shape {
    double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    @Override
    public double area() {
        return Math.PI * radius * radius;
    }
}

public class Rectangle extends Shape {
    double width, height;

    public Rectangle(double width, double height) {
        this.width  = width;
        this.height = height;
    }

    @Override
    public double area() {
        return width * height;
    }
}

public class Main {
    public static void main(String[] args) {
        Shape c = new Circle(5);
        Shape r = new Rectangle(4, 6);

        c.printArea();   // Area: 78.53981633974483
        r.printArea();   // Area: 24.0
    }
}

Encapsulation vs Abstraction — The Difference

These two concepts are often confused. Here is the key distinction:

  • Encapsulation = hiding the data (using private + getters/setters). The concern is who can access the data.
  • Abstraction = hiding the implementation (using abstract classes or interfaces). The concern is what details the user needs to see.
ℹ️ You cannot create an object of an abstract class with new Shape(). An abstract class is meant to be a template — you always instantiate one of its concrete subclasses.
📜 History: Abstraction as a computing concept was formalised by Dijkstra and Parnas in the 1970s with the idea of "information hiding." They argued that the less a module knows about the internal details of other modules, the better. Java's private, public, and protected keywords are direct implementations of this 50-year-old principle.
