Chapter 15: Polymorphism

Polymorphism means "many forms" — the same method call can produce different behaviour depending on which object actually receives it.

The Real-World Idea

Imagine a universal TV remote. It has a single "volume up" button. Whether the TV is a Samsung, a Sony, or an LG, pressing that button works correctly for each brand — each TV responds in its own way to the same command. The remote doesn't need to know the brand; it just presses the button.

That is polymorphism: one interface (the button), many forms of response (brand-specific logic).

Two Kinds of Polymorphism in Java

1. Compile-Time Polymorphism — Method Overloading

Method overloading means having multiple methods with the same name but different parameter lists in the same class. Java decides which version to call at compile time based on the arguments you pass. 

public class Calculator {
    // Add two ints
    public int add(int a, int b) {
        return a + b;
    }

    // Add three ints
    public int add(int a, int b, int c) {
        return a + b + c;
    }

    // Add two doubles
    public double add(double a, double b) {
        return a + b;
    }

    public static void main(String[] args) {
        Calculator calc = new Calculator();
        System.out.println(calc.add(2, 3));         // 5
        System.out.println(calc.add(1, 2, 3));      // 6
        System.out.println(calc.add(1.5, 2.5));     // 4.0
    }
}

🔍 How This Code Works

  • public int add(int a, int b), public int add(int a, int b, int c), public double add(double a, double b) — three methods share the name add but differ in parameter count and types; Java picks the right one at compile time
  • calc.add(2, 3) — two int arguments match the first signature → returns 5
  • calc.add(1, 2, 3) — three int arguments match the second signature → returns 6
  • calc.add(1.5, 2.5) — two double literals match the third signature → returns 4.0
  • This is compile-time polymorphism: the decision of which method to call is made by the compiler, not at runtime

2. Runtime Polymorphism — Method Overriding

Method overriding (from Chapter 14) produces runtime polymorphism. Java decides which version of an overridden method to run at runtime, based on the actual object type — not the reference type.

The key insight: you can store a child object inside a parent-type reference variable: 

Animal a = new Dog();   // Dog IS-A Animal, so this is legal
a.speak();              // calls Dog's speak(), not Animal's

Polymorphism in Practice — Shapes Example

public class Shape {
    public void draw() {
        System.out.println("Drawing a generic shape.");
    }
}

public class Circle extends Shape {
    @Override
    public void draw() {
        System.out.println("Drawing a circle ⬤");
    }
}

public class Rectangle extends Shape {
    @Override
    public void draw() {
        System.out.println("Drawing a rectangle ▬");
    }
}

public class Triangle extends Shape {
    @Override
    public void draw() {
        System.out.println("Drawing a triangle ▲");
    }
}

public class Main {
    public static void main(String[] args) {
        // Parent-type array holding different child objects
        Shape[] shapes = {
            new Circle(),
            new Rectangle(),
            new Triangle()
        };

        // One loop — three different behaviours
        for (Shape s : shapes) {
            s.draw();
        }
    }
}

🔍 How This Code Works

  • Shape[] shapes = { new Circle(), new Rectangle(), new Triangle() }; — an array typed as the parent class (Shape) holds child objects; this is called upcasting and is always safe
  • for (Shape s : shapes) — the loop variable is typed as Shape, but at runtime each element is actually a Circle, Rectangle, or Triangle
  • s.draw(); — the same method call on each iteration; Java's JVM checks the actual object type at runtime and invokes the correct overriding draw() — this is dynamic dispatch
  • @Override public void draw() in each subclass — each subclass replaces the parent's generic draw() with its own specific output
  • Adding a new shape (e.g., Pentagon) requires no changes to the loop — the polymorphic call handles it automatically

Output:

Drawing a circle ⬤
Drawing a rectangle ▬
Drawing a triangle ▲

The for loop calls s.draw() each iteration. Even though s is typed as Shape, Java looks at the actual object (Circle, Rectangle, Triangle) and calls the correct overridden version. This is called dynamic dispatch.

Why Is This Useful?

Polymorphism makes code flexible and extensible. The loop above never needs to change even if you add a Pentagon class later — just add it to the array and the loop handles it automatically. You write code against the parent type (Shape) and plug in any child at runtime.

💡 Tip: Polymorphism is the mechanism that makes Java frameworks and libraries so powerful. When a method accepts a List parameter, it works with ArrayList, LinkedList, or any other List implementation automatically.
🔍 Fun Fact: The word "polymorphism" comes from Greek: "poly" (many) + "morphe" (form). Runtime polymorphism in Java works through a mechanism called "virtual dispatch" — the JVM looks up the actual object type at runtime and calls the correct method. This lookup takes nanoseconds.
⚠️ Common mistake: Confusing overloading (same class, different parameters — compile-time) with overriding (parent vs child, same signature — runtime). They are different mechanisms despite similar-sounding names.
📜 History: Runtime polymorphism was controversial in the 1990s. C++ had it but it was optional (the virtual keyword). Java made all instance methods virtual by default. Critics argued it was too slow. JVMs proved them wrong with JIT (Just-In-Time) compilation, which optimises virtual calls at runtime.
ℹ️ When you use a parent-type reference to hold a child object, you can only call methods that are defined on the parent type. To access child-specific methods, you need to cast: ((Circle) s).getRadius(). Use casting sparingly.
🌍 Real World: Java's List interface is the classic polymorphism example from production code. A method written to accept List<String> works equally with ArrayList, LinkedList, and CopyOnWriteArrayList — you can swap implementations without touching the calling code. Spring Boot and most Java frameworks rely on this pattern constantly.

🧠 Multiple Choice Questions

1. What is runtime polymorphism in Java?

  • A) Method overloading resolved at compile time
  • B) The ability of an object to take many forms, with the correct method chosen at runtime
  • C) Converting types at runtime
  • D) Running multiple programs simultaneously
Show Answer

✅ B) The ability of an object to take many forms, with the correct method chosen at runtime — When a parent reference holds a child object, Java calls the child's overriding method at runtime.

2. Which annotation should you use when overriding a method?

  • A) @Override
  • B) @Overwrite
  • C) @Replace
  • D) @Super
Show Answer

✅ A) @Override — @Override tells the compiler to verify that you're actually overriding a parent method, catching typos in method signatures.

3. For method overriding, the overriding method must have:

  • A) A different return type
  • B) More parameters
  • C) The same name, same parameters, same or covariant return type
  • D) The word "override" in its name
Show Answer

✅ C) The same name, same parameters, same or covariant return type — Overriding redefines the behavior of an inherited method with the exact same signature.

4. What is dynamic dispatch?

  • A) Sending messages over a network
  • B) The JVM mechanism that calls the correct overriding method based on the actual object type at runtime
  • C) A multithreading technique
  • D) Dispatching constructors
Show Answer

✅ B) The JVM mechanism that calls the correct overriding method based on the actual object type at runtime — This is what makes polymorphism work: the JVM checks the actual object type, not the reference type.
