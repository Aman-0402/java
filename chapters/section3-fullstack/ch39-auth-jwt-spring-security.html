Chapter 39: Auth — JWT + Spring Security

Secure your Spring Boot API so only logged-in users can access protected endpoints.

Why Authentication?

Right now anyone can call DELETE /api/todos/1 and destroy data. Authentication prevents this — only users who prove their identity (login) can access protected endpoints.

📜 History: JWT (JSON Web Token) was standardized as RFC 7519 in 2015 by the IETF. Before JWT, most web apps used server-side sessions — the server stored session data in memory or a database, and the client just sent a session ID cookie. JWTs moved session storage to the client (the token itself contains user data), reducing server memory usage and enabling stateless, horizontally-scalable architectures.

What is JWT?

JWT stands for JSON Web Token. After a user logs in, the server creates a signed token and sends it to the client. The client sends this token with every future request. The server verifies the signature to confirm the token is genuine — no database lookup needed.

# A JWT looks like this (3 parts separated by dots):
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMSIsImlhdCI6MTY5MDAwMH0.XyZ_signature

# Part 1: Header (algorithm)
# Part 2: Payload (user data — not secret, just encoded)
# Part 3: Signature (proves it was made by your server)
🔍 Fun Fact: A JWT's signature uses HMAC-SHA256 — a cryptographic algorithm. HMAC stands for Hash-based Message Authentication Code. The math behind it involves modular arithmetic and was designed by Mihir Bellare and Ran Canetti in 1996. When your app validates a JWT, it's running a cryptographic proof that was peer-reviewed by cryptographers. You get military-grade security from a 3-line code snippet.
⚠️ The JWT payload is Base64-encoded, NOT encrypted. Never put passwords or sensitive data inside a JWT. It can be decoded by anyone.

Step 2: JWT Utility Class

import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.security.Keys;
import java.util.Date;
import javax.crypto.SecretKey;

public class JwtUtil {

    // Use a long, random secret key in production — store in environment variable
    private static final String SECRET = "mySecretKey12345mySecretKey12345";
    private static final SecretKey KEY = Keys.hmacShaKeyFor(SECRET.getBytes());
    private static final long EXPIRY = 1000 * 60 * 60 * 24; // 24 hours

    public static String generateToken(String username) {
        return Jwts.builder()
            .subject(username)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + EXPIRY))
            .signWith(KEY)
            .compact();
    }

    public static String extractUsername(String token) {
        return Jwts.parser()
            .verifyWith(KEY)
            .build()
            .parseSignedClaims(token)
            .getPayload()
            .getSubject();
    }

    public static boolean isValid(String token) {
        try {
            extractUsername(token);
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}

🔍 How This Code Works

  • Keys.hmacShaKeyFor(SECRET.getBytes()) — creates a cryptographic signing key from the secret string; HMAC-SHA256 requires a key of at least 256 bits (32 characters)
  • private static final long EXPIRY = 1000 * 60 * 60 * 24 — token lifetime in milliseconds; 1000ms × 60s × 60min × 24h = 24 hours
  • Jwts.builder().subject(username).issuedAt(...).expiration(...).signWith(KEY).compact() — builder chain that constructs the JWT; compact() serializes it to the dot-separated string
  • Jwts.parser().verifyWith(KEY).build().parseSignedClaims(token) — validates the token's signature against the same secret key; throws an exception if tampered or expired
  • .getPayload().getSubject() — extracts the sub claim from the payload, which holds the username set during token creation
  • isValid() — wraps parsing in a try-catch; returns false for expired, tampered, or malformed tokens instead of throwing

Step 3: Login Endpoint

@RestController
@CrossOrigin(origins = "*")
public class AuthController {

    @PostMapping("/api/login")
    public ResponseEntity<?> login(@RequestBody LoginRequest req) {
        // In real apps: check username/password against database
        if ("user".equals(req.getUsername()) && "password".equals(req.getPassword())) {
            String token = JwtUtil.generateToken(req.getUsername());
            return ResponseEntity.ok(Map.of("token", token));
        }
        return ResponseEntity.status(401).body("Invalid credentials");
    }
}

// Simple DTO
class LoginRequest {
    private String username;
    private String password;
    // getters and setters
    public String getUsername() { return username; }
    public void setUsername(String u) { this.username = u; }
    public String getPassword() { return password; }
    public void setPassword(String p) { this.password = p; }
}

Step 4: Send Token from React

// After login, store the token
const login = async () => {
  const res = await fetch('http://localhost:8080/api/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username: 'user', password: 'password' })
  });
  const data = await res.json();
  localStorage.setItem('token', data.token);
};

// Send token with protected requests
const getTodos = async () => {
  const token = localStorage.getItem('token');
  const res = await fetch('http://localhost:8080/api/todos', {
    headers: { 'Authorization': 'Bearer ' + token }
  });
  return res.json();
};
🌍 Real World: Spotify uses JWT for its API authentication. When you connect a third-party app to Spotify (like a playlist manager), Spotify issues a JWT. The token contains your permissions (read playlists, modify library) encoded in the payload. The third-party app sends this token with every API request — Spotify validates the signature without hitting a database.
💡 localStorage is simple for learning but has XSS risks in production. For production apps consider using httpOnly cookies instead.
ℹ️ This chapter covers the core JWT concepts. Full Spring Security configuration (JWT filter, SecurityFilterChain) is an advanced topic — many production teams use libraries like Spring Security's OAuth2 resource server support or third-party auth providers (Auth0, Firebase Auth) to avoid writing JWT filters manually.

🧠 Multiple Choice Questions

1. What does JWT stand for?

  • A) Java Web Token
  • B) JSON Web Tool
  • C) JSON Web Token
  • D) JavaScript Web Transfer
Show Answer

✅ C) JSON Web Token — A JWT is a compact, URL-safe token containing a JSON payload that can be verified via a digital signature.

2. What are the three parts of a JWT separated by dots?

  • A) Username, password, expiry
  • B) Header, Payload, Signature
  • C) ID, data, checksum
  • D) Type, version, content
Show Answer

✅ B) Header, Payload, Signature — Header contains algorithm info, Payload contains claims (user data), Signature verifies authenticity.

3. How does a server validate a JWT without hitting the database?

  • A) It can't — database lookup is always required
  • B) It decrypts the token with a private key
  • C) It re-computes the signature using the secret key and compares with the token's signature
  • D) It calls the auth server
Show Answer

✅ C) It re-computes the signature using the secret key and compares with the token's signature — If the signatures match, the token is valid and hasn't been tampered with. No database needed.

4. What HTTP header is used to send a JWT to the server?

  • A) X-Auth-Token
  • B) Cookie
  • C) Authorization: Bearer [token]
  • D) JWT-Token
Show Answer

✅ C) Authorization: Bearer [token] — The standard format is Authorization: Bearer eyJhbGci.... The server extracts and validates the token from this header.
