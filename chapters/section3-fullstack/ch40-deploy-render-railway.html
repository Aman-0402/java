Chapter 40: Deploy to Cloud

Take your full-stack Java application live on the internet — free hosting for both backend and frontend.

What is Deployment?

Right now your app only runs on your laptop. Deployment means uploading it to a cloud server so anyone with the URL can access it — 24/7, without your laptop being on.

We will deploy:

  • Spring Boot backend → Render.com (free tier)
  • React frontend → Netlify (free tier)
📜 History: Cloud hosting was revolutionized by Amazon Web Services (AWS) in 2006 when Amazon launched S3 (storage) and EC2 (virtual servers). Before AWS, companies bought physical servers, installed them in data centers, and managed everything manually. AWS let developers rent servers by the hour. This enabled startups to scale instantly without capital investment. Render and Railway are modern successors — even simpler than AWS.

Step 1: Prepare Spring Boot for Production

Switch from H2 (in-memory) to a real database. Render provides a free PostgreSQL database. Add the PostgreSQL driver:

<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <scope>runtime</scope>
</dependency>

Update application.properties to use an environment variable for the database URL (never hardcode credentials):

spring.datasource.url=${DATABASE_URL}
spring.datasource.driver-class-name=org.postgresql.Driver
spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect
spring.jpa.hibernate.ddl-auto=update

Step 2: Build the JAR

# Package your Spring Boot app into a single executable JAR
mvn package -DskipTests

# The JAR is created at:
target/your-app-name-0.0.1-SNAPSHOT.jar

# Test it locally first:
java -jar target/your-app-name-0.0.1-SNAPSHOT.jar
🔍 Fun Fact: Netflix runs entirely on AWS (Amazon Web Services). They have over 1,000 microservices deployed across multiple AWS regions. In 2011, Netflix famously had a major AWS outage on Christmas Eve that took down their service for hours. This led them to build "Chaos Monkey" — a tool that randomly kills production servers to force engineers to build resilient systems.
💡 -DskipTests skips tests during packaging to speed up the build. Always run tests separately with mvn test before deploying.

Step 3: Deploy Backend to Render

  1. Push your Spring Boot project to a GitHub repository
  2. Go to render.com → Sign up → New → Web Service
  3. Connect your GitHub repo
  4. Settings:
    • Build Command: mvn package -DskipTests
    • Start Command: java -jar target/*.jar
    • Environment: Java
  5. Create a free PostgreSQL database on Render → copy the Internal Database URL
  6. Add environment variable: DATABASE_URL = the PostgreSQL URL
  7. Click Deploy — Render builds and starts your app
⚠️ Render's free tier spins down after 15 minutes of inactivity and takes ~30 seconds to cold-start. This is normal for the free tier. Paid tier keeps it always-on.

Step 4: Deploy Frontend to Netlify

  1. In your React project, update the API base URL to your Render backend URL: 
    const API_URL = 'https://your-app.onrender.com';
  2. Build React for production: npm run build
  3. Go to netlify.com → Sign up → New site → Import from Git
  4. Connect your GitHub repo
  5. Settings:
    • Build command: npm run build
    • Publish directory: build (or dist if using Vite)
  6. Click Deploy site — Netlify builds and gives you a live URL

🌍 Real World: When you deploy to Render, your Spring Boot JAR runs inside a Docker container on a Linux virtual machine in a data center. Render handles the Linux setup, SSL certificates, load balancing, and auto-restarts. What used to require a dedicated DevOps engineer — server provisioning, certificate management, deployment pipelines — now takes 5 minutes for a single developer.
ℹ️ Railway.app is an alternative to Render — also has a free tier, often simpler to configure. Both work well for learning projects.
